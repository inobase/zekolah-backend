## What Needs Testing

T5.4 requires verifying that a user who holds **multiple active roles** (possibly across multiple schools and/or academic years) receives correctly scoped data when querying endpoints. Each role context should filter data appropriately.

**Example scenarios to test:**
1. User with `teacher` role at School 1 + `teacher` role at School 2 → when requesting with `x-school-id=1`, they see only School 1 data; with `x-school-id=2`, they see only School 2 data
2. User with `teacher` role at School 1 + `staff` role at School 1 → both roles resolved, data filtered by school context
3. User with school-wide role + academic-year-specific role → specific role should take priority
4. Global role (school_id = NULL) → should allow access to all schools OR be restricted by context depending on policy

## Current Architecture Analysis

### How Multi-Role Works Today

#### 1. Role Assignment Storage
- `user_roles` table stores multiple roles per user with unique constraint on `[user_id, role_id, school_id, academic_year_id]`
- Each entry can have different `school_id` and `academic_year_id` values

#### 2. Role Resolution (`UserRoleRepository.findActive()`)
```sql
SELECT r.name as role, ur.school_id, ur.academic_year_id, ur.is_active
FROM user_roles ur
JOIN roles r ON r.id = ur.role_id
WHERE ur.user_id = ? AND ur.is_active = 1
  AND (
    (ur.school_id = ? AND ur.academic_year_id = ?)
    OR (ur.school_id = ? AND ur.academic_year_id IS NULL)
    OR (ur.school_id IS NULL AND ur.academic_year_id IS NULL)
  )
```

**Deduplication logic (priority-based):**
1. Exact school + academic year match (most specific) — e.g., `(school_id=1, academic_year_id=3)`
2. School-wide match — e.g., `(school_id=1, academic_year_id=NULL)`
3. Global — e.g., `(school_id=NULL, academic_year_id=NULL)`

**Deduplication is by role NAME only**, keeping the first match found. Priority order: exact > school > global.

#### 3. Login Token Payload (`AuthService.login()`)
JWT token includes:
```typescript
{
  id: user.id,
  email: user.email,
  school_id: ctxSchoolId,     // highest priority school from all roles
  academic_year_id: ctxAYId,  // highest priority academic year from all roles
  roles: [{ role, school_id, academic_year_id }, ...],  // ALL resolved roles
}
```

**Context selection logic** (simple greedy first-match):
```typescript
for (const role of allRoles) {
  if (role.school_id != null && role.academic_year_id != null) {
    ctxSchoolId = role.school_id
    ctxAYId = role.academic_year_id
    break
  }
  if (ctxSchoolId == null && role.school_id != null) {
    ctxSchoolId = role.school_id
  }
}
```
This picks the **first** role that has BOTH `school_id` and `academic_year_id`. No priority sorting applied before selection — depends on DB query order.

#### 4. Request-time Context Resolution (`app.ts` authenticate middleware)
For **every authenticated request**:
```typescript
// Priority: header > JWT payload > null
const ctxSchoolId = (schoolId ?? request.user.school_id ?? null)
const ctxAYId = (academicYearId ?? request.user.academic_year_id ?? null)
const resolvedRoles = await roleResolver.resolve(request.user.id, ctxSchoolId, ctxAYId)
request.activeSchoolId = ctxSchoolId
request.activeAcademicYearId = ctxAYId
request.resolvedRoles = resolvedRoles
```

Key: `roleResolver.resolve()` queries `user_roles` using the current `ctxSchoolId` + `ctxAYId`, applying the priority-based deduplication.

#### 5. Controller Behavior
Every controller uses `req.activeSchoolId` as the **single source of truth** for school isolation:
```typescript
const filter = { ...query, school_id: req.activeSchoolId ?? undefined }
```

**Critical:** Controllers do NOT check `req.resolvedRoles` for role-based data filtering. All filtering is purely school-scoped. A user with multiple roles at different schools simply sees data filtered by the **currently active school context**.

## Potential Issues / Gaps Found

### Issue 1: Role Resolution May Not Honor Requested School Context Correctly

The `findActive()` query uses NULL-coalescing logic with `OR` conditions:
```sql
(ur.school_id = ? AND ur.academic_year_id = ?)
OR (ur.school_id = ? AND ur.academic_year_id IS NULL)
OR (ur.school_id IS NULL AND ur.academic_year_id IS NULL)
```

This query will match roles at the specified school AND global roles. But the deduplication keeps only the **first** role per name. If the DB returns roles in an unpredictable order, the priority may not be consistent.

**Risk:** If a user has roles at both school 1 and school 2, switching `x-school-id` header might or might not correctly swap the resolved roles depending on DB return order.

### Issue 2: Login Context Selection Is Unstable

The greedy loop in `AuthService.login()` iterates `allRoles` in whatever order the DB returns them. There's no `ORDER BY` or priority sorting applied. This means the `school_id` and `academic_year_id` stored in the JWT token may vary between logins if DB query order is non-deterministic.

**Risk:** A user logged in at school 1 might get JWT with school 2 as default context on the next login.

### Issue 3: `findActive()` Deduplication Might Drop Valid Distinct Roles

The deduplication `seen.has(row.role)` is by role NAME only. If a user has:
- `teacher` at school 1, academic year 2024
- `teacher` at school 1, academic year 2025

Both have the same role name. The query returns both, but deduplication keeps only the first one (whichever the DB returns first).

**Risk:** Academic-year-scoped roles are silently collapsed.

### Issue 4: Controllers Only Use `req.activeSchoolId`, Not `req.resolvedRoles`

All controller filtering is purely based on `req.activeSchoolId`. There is **no role-based data scoping**. A user with `teacher` role sees all students in their school, regardless of whether they are also a `staff` role. Conversely, a `super_admin` role doesn't grant cross-school access if `req.activeSchoolId` is constrained.

**Implication:** Role-based permissions (e.g., teacher can only see their own students) are **NOT implemented** in controllers today. The multi-role system only affects **which school's data** is visible, not **what entities within that school** are accessible.

### Issue 5: `resolve()` Always Applies School Filter

When `schoolId` parameter is `null`, the `findActive()` query becomes:
```sql
WHERE (
  (ur.school_id = NULL AND ur.academic_year_id = NULL)
  OR (ur.school_id = NULL AND ur.academic_year_id IS NULL)
  OR (ur.school_id IS NULL AND ur.academic_year_id IS NULL)
)
```
Which matches only global roles (where `school_id IS NULL`). This is correct but could be confusing — if a user has no global roles, they'll get an empty `resolvedRoles` array.

## Test Plan for T5.4

### Setup
1. Create two schools (School A and School B)
2. Create a user with:
   - `teacher` role at School A (with academic year X)
   - `teacher` role at School B (with academic year X)
3. Optionally add a third role: `staff` at School A (school-wide, no academic year)

### Test Cases

| # | Scenario | Expected Behavior |
|---|----------|-------------------|
| TC1 | User logs in (no header) → check JWT payload | JWT should contain a deterministic `school_id` and `academic_year_id` (prefer highest specificity) |
| TC2 | User sends request with `x-school-id=A` | `req.activeSchoolId` = A → sees only School A students/teachers/classes |
| TC3 | User sends request with `x-school-id=B` | `req.activeSchoolId` = B → sees only School B students/teachers/classes |
| TC4 | User sends request with `x-school-id=A` + `x-academic-year-id=X` | Resolves teacher role with specific AY context; data scoped to School A + AY X |
| TC5 | User sends request with `x-school-id=C` (user has no role) | Should return empty results or 404 (no matching school) |
| TC6 | Login again → verify JWT payload is deterministic | Same `school_id`/`academic_year_id` on every login |
| TC7 | User with multiple roles at same school (teacher + staff) | Both roles appear in `resolvedRoles`; data filtered by school only |
| TC8 | Verify list endpoints respect school context | `GET /students` returns only students from active school |
| TC9 | Verify getById endpoints respect school context | `GET /students/:id` returns 404 if student belongs to different school |
| TC10 | Verify create endpoints auto-fill correct school_id | `POST /students` creates student with `req.activeSchoolId` |

### Verification Points
- `req.activeSchoolId` correctly reflects the chosen school context
- `req.resolvedRoles` contains the correct resolved roles for the chosen context
- List/getById/create endpoints all respect the school scope
- Switching `x-school-id` header consistently changes the data scope
- Login returns stable, deterministic context in JWT

## Implementation Plan: Per-File Testing Strategy

Instead of writing one monolithic test file, T5.4 should be tested **per-file** following the existing pattern (e.g., `cross-school-leakage.test.ts` was built incrementally). Each test file focuses on a specific module and verifies multi-role behavior within that module's scope.

### Suggested File Structure

```
tests/
├── multi-role-context.test.ts          # Core: role resolution, JWT payload, header swapping
├── multi-role-students.test.ts         # Student endpoints with multi-role user
├── multi-role-teachers.test.ts         # Teacher endpoints with multi-role user
├── multi-role-classes.test.ts          # Class endpoints with multi-role user
├── multi-role-subjects.test.ts         # Subject endpoints with multi-role user
├── multi-role-grades.test.ts           # Grade endpoints with multi-role user
├── multi-role-attendance.test.ts       # Attendance endpoints with multi-role user
├── multi-role-submissions.test.ts      # Submission endpoints with multi-role user
└── multi-role-academic-years.test.ts   # AcademicYear endpoints with multi-role user
```

### Phase Breakdown

**Phase 1 — Core Context Resolution** (`multi-role-context.test.ts`)
- T6.1: User with teacher roles at School A and School B → JWT contains stable context
- T6.2: `x-school-id=A` resolves to School A roles, `x-school-id=B` resolves to School B roles
- T6.3: `x-school-id=C` (no role) returns empty/no results
- T6.4: Login determinism — same context returned on repeated logins
- T6.5: `resolvedRoles` array contains correct roles for current context

**Phase 2 — Module-Specific Multi-Role Scoping** (one file per module)
Each module file follows the same pattern:
- T6.6a: `GET /students` returns only School A data when `x-school-id=A`
- T6.6b: `GET /students` returns only School B data when `x-school-id=B`
- T6.6c: `GET /students/:id` returns 404 for School B data when context is School A
- T6.6d: `POST /students` creates student with correct `school_id`
- T6.6e: `PATCH /students/:id` cross-school returns 404

This approach lets us:
- Incrementally build and verify each module
- Easily identify which module fails or passes
- Follow the same pattern used successfully in T5.2 (cross-school-leakage)
- Keep test files focused and maintainable (~20-30 lines per test)

### Recommended Order
1. `multi-role-context.test.ts` first — establishes baseline context switching
2. Then add module-specific files one at a time (students, teachers, classes, etc.)
3. Finally run all together to ensure no cross-contamination between test schools
