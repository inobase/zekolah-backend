# v1.0.4 — Multi-Tenant Scoped Role System (Phase 1–5)

## Tanggal
2026-07-16

## Summary
Mengimplementasikan **multi-tenant scoped role system** yang menggantikan `users.role` (flat role per-user) menjadi `user_roles` table (scoped per user + role + school + academic_year). Sistem ini mendukung:

- **Multi-school teachers**: Satu guru bisa mengajar di beberapa sekolah
- **Scoped role assignments**: Role bisa bersifat global (cross-school), school-specific, atau academic_year-specific
- **Context switching via headers**: Header `X-School-ID` dan `X-Academic-Year-ID` memungkinkan admin switching konteks sesi
- **Fallback backwards-compatible**: Legacy `users.role` masih dipakai sebagai fallback terakhir di middleware

## Arsitektur Baru

### Tabel
| Table | Keterangan |
|-------|------------|
| `roles` | Definition role: `super_admin`, `admin`, `staff`, `teacher`, `student` (5 seeds) |
| `user_roles` | Scoped assignment: `user_id` + `role_id` + `school_id` (nullable) + `academic_year_id` (nullable) |
| `users.role` | ~~Kolom legacy~~ — **DI-DROP** di Phase 5 |

### Scoping Rules
- `school_id = NULL` = **cross-school** (global, berlaku semua sekolah)
- `academic_year_id = NULL` = **all academic years**
- Priority saat resolve role: exact school+year > school-wide > cross-school

### Role Resolution Flow (setiap request)
```
User request
  → JWT verify (app.authenticate)
  → Parse X-School-ID / X-Academic-Year-ID headers (fallback ke JWT payload)
  → Resolve roles via user_roles JOIN roles
  → Inject roles + active context ke req.user + req object
  → Route handler receives augmented user with roles array
```

## Perubahan per Phase

### Phase 1 — Schema & Data Migration
**File:**
- `migrations/015_roles.ts` — Create `roles` table + seed 5 role
- `migrations/016_user_roles.ts` — Create `user_roles` table + indexes + migrate existing data

**Migration Data:**
- Teachers → `user_roles` dengan `role_id=teacher` + `school_id=teachers.school_id`
- Students → `user_roles` dengan `role_id=student` + `school_id=students.school_id`
- Other users → `user_roles` dengan `role_id=(users.role)` + `school_id=NULL` (global)

### Phase 2 — Helpers, Types & Repositories
**Files:**
- `src/models/interfaces/RoleInterfaces.ts` — TypeScript interfaces: `Role`, `UserRole`, `ResolvedUserRole`, `AugmentedUser`
- `src/repositories/role.repository.ts` — `findByName`, `findById`, `listAll`
- `src/repositories/userRole.repository.ts` — `findByUserId`, `findActive` (raw SQL UNION), `insert`, `deactivate`, `delete`
- `src/utils/roleResolver.ts` — `RoleResolver` class: `resolve()`, `hasAnyRole()`, `hasAllRoles()`, `buildAugmentedUser()`
- `src/middlewares/requireRole.ts` — Factory middleware: accepts `string`, `string[]`, or predicate `(roles) => boolean`

### Phase 3 — Auth Middleware Integration
**File:**
- `src/app.ts` — Enhanced `app.authenticate` decorator:
  1. Parse `X-School-ID` + `X-Academic-Year-ID` headers
  2. Resolve roles via `RoleResolver` (most specific first)
  3. Inject `roles`, `activeSchoolId`, `activeAcademicYearId` ke `request.user` AND `request`

### Phase 4 — Routes & Controllers Audit
**Audit Result:** Tidak ada blanket refactor dilakukan.
- 14 routes file + 13 controllers diaudit → semua endpoint sudah dilindungi `app.authenticate`
- Tidak ada `req.user.role` legacy usage di mana pun (hanya fallback di `requireRole.ts`)
- Role-specific guards (`requireRole([...])`) akan ditambahkan **per-fitur** saat business flow sudah mapan
- Semua endpoint deferred (T4.1-T4.6) sebagai audit-only

### Phase 5 — Cleanup (Deprecation)
**Migration:**
- `migrations/017_drop_users_role_column.ts` — DROP COLUMN `role` dari `users` table

**Code Removal (role field dropped dari 15+ files):**
- **Validators:** `RegisterSchema`, `CreateUserSchema`, `UserFilterSchema` — removed `role` field
- **Interfaces:** `AuthUser`, `SafeUser`, `AugmentedUser` — removed `role` field
- **Services:** `AuthService.register()`, `login()`, `UserService.list()`, `create()` — removed role parameter/field
- **Repositories:** `UserRepository.create()` — removed role from insert
- **Middleware:** `requireRole.ts` — removed legacy fallback `user.role ? [...]`
- **App:** `app.ts` FastifyJWT declaration — removed `role?: string`
- **Utils:** `RoleResolver.buildAugmentedUser()` — removed `legacyRole` parameter
- **Tests:** Removed `role` from 13 test files, updated queries from `users.role` to `user_roles JOIN roles`

## API Impact (Breaking Changes)

### Register Endpoint
```diff
- POST /api/v1/auth/register

+ POST /api/v1/auth/register
  { email, password, name }  // role no longer accepted
```

### User Creation Endpoint
```diff
- POST /api/v1/users
  { email, password, name, role: 'admin' }
  
+ POST /api/v1/users
  { email, password, name }  // role no longer accepted
```

### User Filter Endpoint
```diff
- GET /api/v1/users?role=teacher  // filter removed
+ GET /api/v1/users?search=&status=  // role filter no longer valid
```

### Response Shape
```diff
  { user: { id, email, name, password, token } }
-   user.role          // REMOVED from response
```

| `X-Academic-Year-ID` | Override academic year context untuk role resolution |

## Trade-offs

| Trade-off | Catatan |
|-----------|---------|
| Breaking change: `users.role` DROPPED | Semua client harus update — tidak lagi terima/kirim `role` field. Role management sekarang 100% via `user_roles` table. |
| Raw SQL di `userRole.repository.ts` | `findActive()` menggunakan raw SQL UNION untuk scoping logic. Query builder Knex tidak mendukung kompleksitas ini dengan baik. |
| No role guards in routes yet | Phase 4 audit menemukan semua endpoint sudah protected via `app.authenticate`. Actual `requireRole()` guards akan ditambahkan per-fitur saat business flow matang. |
| Test data isolation | School DELETE test gagal karena FK cascade dari `user_roles` → `users` → cascade ke school. Ini issue pre-existing dari seed data. |
