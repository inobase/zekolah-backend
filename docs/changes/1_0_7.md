# CHANGELOG v1.0.7 — Schedule Tests Bugfixes & SCHEDULE_CONFLICT Hardening

**Date:** 2026-07-19
**Scope:** Backend API — schedule module test fixes, error code mapping, write-endpoint role guards

---

## Summary

This release resolves all 13 failing tests in `tests/schedule.test.ts` (Phase 5 — Schedules) and hardens the schedule module's write endpoints with role-based access control. Key changes:

- **Fixed missing test helper** — `createTeacherForSchool(schoolId, suffix)` was referenced by 13 tests but never defined; implemented inside the `describe` block following the existing `createSchoolAndTeacher` pattern.
- **`SCHEDULE_CONFLICT` mapped to HTTP 409** — `AppError.defaultStatusFor()` was missing the new error code, so double-booking conflicts returned 500 instead of 409.
- **Write endpoints role-guarded** — POST/PATCH/DELETE on `/schedules` now require `admin` or `super_admin` via `requireRole` middleware; teachers and students are blocked from mutations.

All **20/20 schedule tests now pass**.

## Bug Fixes

### Bug 1 — Missing `createTeacherForSchool` helper (13 failures)

**Symptom:** `ReferenceError: createTeacherForSchool is not defined` thrown by 13 tests.

**Root Cause:** The test file ended with the comment `// Helper for creating extra teachers` but the actual function definition was never written. Many tests called this helper expecting it to behave like `createSchoolAndTeacher(schoolId)` — create a teacher scoped to a school with a unique email.

**Fix:** Added `createTeacherForSchool(schoolId: number, suffix: string)` inside the `describe('Schedules')` block so it has access to the shared `knex` instance. The function:
- Cleans up existing user/teacher by email
- Creates a new user + `teachers` row scoped to the given school
- Uses email pattern `teacher_${suffix}_${schoolId}@zet.com` for uniqueness
- Returns `{ teacherEmail, tUid, teacherId }`

**Affected tests (all now passing):**
- POST then GET lists schedules with pagination
- GET filters by class_id
- GET filters by teacher_id
- GET by ID returns schedule with details
- PATCH updates a schedule
- DELETE deletes a schedule
- POST returns 409 for class double-booking
- POST returns 409 for teacher double-booking
- GET /schedules/class/:classId returns schedules for class
- GET /schedules/class/:classId/timetable returns grouped by day
- GET /schedules/conflicts returns conflict list
- TEACHER cannot POST schedules (write)
- Cross-school access returns filtered results

### Bug 2 — `SCHEDULE_CONFLICT` mapped to 500 instead of 409 (2 failures)

**Symptom:** When a POST created a schedule that conflicted with an existing one (class or teacher already booked for the same day/time), the API returned HTTP 500 instead of the expected HTTP 409.

**Root Cause:** The `AppError` type added `SCHEDULE_CONFLICT` as a 409-level error code, but the `defaultStatusFor(code)` switch statement in `src/utils/AppError.ts` was not updated to handle it. The switch fell through to the default `return 500`, so conflicts were reported as internal server errors.

**Fix:** Added an explicit case in `AppError.defaultStatusFor()`:
```typescript
case 'SCHEDULE_CONFLICT':
  return 409;
```

**Affected tests (now passing):**
- POST returns 409 for class double-booking
- POST returns 409 for teacher double-booking

### Bug 3 — Teacher role could POST/PATCH/DELETE schedules (1 failure)

**Symptom:** A request authenticated as a TEACHER successfully created a schedule (returned 201) instead of being rejected with 403 Forbidden.

**Root Cause:** The schedule routes only had `onRequest: [app.authenticate]` on POST/PATCH/DELETE. The `authenticate` decorator verifies that a JWT token is present and valid, but does not enforce role membership. The Phase 5 controller comment said "SCHOOL_ADMIN scope for writes; TEACHER for reads" but the role guard was never actually wired into the routes.

**Fix:** Imported `requireRole` from `src/middlewares/requireRole.ts` and added `preHandler: requireRole(['admin', 'super_admin'])` to all three write endpoints:
- `POST /api/v1/schedules`
- `PATCH /api/v1/schedules/:id`
- `DELETE /api/v1/schedules/:id`

Read endpoints (`GET /schedules`, `GET /schedules/:id`, `GET /schedules/class/:classId`, `GET /schedules/class/:classId/timetable`, `GET /schedules/teacher/:teacherId`, `GET /schedules/conflicts`) remain unguarded so teachers and students can still access their timetables.

**Affected test (now passing):**
- TEACHER cannot POST schedules (write)
