# CHANGELOG v1.0.6 — Multi-Tenant Data Isolation & Role Management API

**Date:** 2026-07-18
**Scope:** Backend API — school context enforcement, role CRUD, JWT payload enhancement, cross-school isolation, multi-role scoped data

---

## Summary

This release implements multi-tenant data isolation and role management across all API endpoints. Key improvements include:

- **Enforced school context injection** — `req.activeSchoolId` auto-injected into all 12+ controllers (students, teachers, classes, subjects, assignments, grades, attendance, submissions, academic years), eliminating reliance on frontend query params.
- **Cross-school isolation** — all 12 repositories now filter by `school_id`; 25+ integration tests verify school A cannot access school B data across all entity types.
- **Role management API** — 6 new endpoints for assign, list, activate/deactivate, and delete user roles, plus self-service `/me/roles` and `/me/context` endpoints.
- **JWT payload enhancement** — login/register responses now include `school_id` + `academic_year_id` context and resolved roles array.
- **Multi-role scoped data (T5.4)** — 66 tests verifying users with multiple roles receive correctly scoped data across 8 entity types (context, students, teachers, classes, subjects, grades, attendance, submissions).
- **SchoolController role-based filtering** — admin/super_admin sees all schools; school-scoped users see only their assigned schools.
- **userRole routes OpenAPI migration** — all 6 role endpoints now use `ZodTypeProvider` with full Swagger schemas (tags, summaries, security, params, body, response).

## Multi-Role Scoping Resolution

Role priority is deterministic (highest specificity first):

1. **Exact school + academic year** — role scoped to specific school and AY
2. **School-wide** — role scoped to school only (AY = null)
3. **Global/cross-school** — role scoped to no school (school_id = null)

Priority scores: `3` (exact) > `2` (school-wide) > `1` (global-null) > `0` (fully-global).

## Cross-School Isolation Verification

25 integration tests in `tests/cross-school-leakage.test.ts` verify:

| Entity | School A → School B Access | Enforcement |
|--------|--------------------------|-------------|
| Students | Forbidden (404) | Repository `school_id` filter |
| Teachers | Forbidden (404) | Repository `school_id` filter |
| Classes | Forbidden (404) | Repository `school_id` filter |
| Subjects | Forbidden (404) | Repository `school_id` filter |
| Grades | Forbidden (404) | JOIN via students (`school_id` filter) |
| Attendance | Forbidden (404) | JOIN via students (`school_id` filter) |
| Submissions | Forbidden (404) | JOIN via students (`school_id` filter) |
| Academic Years | Forbidden (404) | Repository `school_id` filter |
| Assignments | Skipped | No `school_id` column on assignments table |

Changes from user to school-scoped users result in 404 NOT_FOUND (not 403) to prevent existence enumeration.

## Backward Compatibility

- **Breaking:** None. School context injection is additive — controllers that previously relied on frontend query params now auto-inject from authenticated session. Frontend sending explicit `x-school-id` headers still works (priority: header > JWT payload).
- **New:** 404 instead of 403 when accessing another school's resources (security improvement — prevents resource enumeration).
- **New:** Role management endpoints under `/api/v1/user-roles/` prefix.
- **New:** JWT payload now includes `school_id`, `academic_year_id`, and `roles` array on login.

---

## Known Limitations

1. **Assignments lack `school_id` column** — cross-school isolation for assignments is enforced indirectly via class ownership (skipped in tests).
2. **Schedule module does not exist yet** — T5.4 Phase 9 for schedule scoping is blocked.
3. **SchoolController role-based filtering** — only applies to `list` endpoint. `getById`, `update`, `delete` still accessible to any authenticated user (future phase).
