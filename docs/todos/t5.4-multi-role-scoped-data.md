## Critical Bugs Found During Analysis (Must Fix Before Testing)

### BUG-1: Role Resolution Deduplication Drops AY-Scoped Roles 🔴
**File:** `src/repositories/userRole.repository.ts` — `findActive()`

**Problem:** Deduplication by role NAME only — if user has `teacher` at school 1 AY 2024 and `teacher` at school 1 AY 2025, only one is kept.

```typescript
// Current (buggy):
const seen = new Set<string>()
return rows.filter((row) => {
  if (seen.has(row.role)) return false
  seen.add(row.role)
  return true
})
```

**Fix:** Deduplicate by `(role, school_id, academic_year_id)` composite key, not just role name alone.

### BUG-2: Login Context Selection Non-Deterministic 🔴
**File:** `src/services/auth.service.ts` — `login()`

**Problem:** Greedy iteration over `allRoles` picks first matching role, but DB query order is unstable. Different logins may yield different `school_id`/`academic_year_id` in JWT.

```typescript
// Current (buggy):
for (const role of allRoles) {
  if (role.school_id != null && role.academic_year_id != null) {
    ctxSchoolId = role.school_id
    ctxAYId = role.academic_year_id
    break  // First match, not necessarily highest priority
  }
  ...
}
```

**Fix:** Sort roles by specificity (exact AY > school-wide > global) BEFORE picking the first match.

### BUG-3: Controller Role-Based Filtering Not Implemented 🟡
**File:** All controllers (e.g., `src/controllers/student.controller.ts`)

**Problem:** All controllers filter only by `req.activeSchoolId`. There is NO role-based filtering. A `teacher` sees ALL students in their school, not just their assigned ones. A `student` role can still access `/students` endpoint without error (returns all students if school matches).

**Decision for T5.4:** Phase 4 scope is **multi-role school context switching**, NOT role-based entity filtering. We test that:
- Multi-role user gets correct `school_id` context based on `x-school-id` header
- Each school context correctly scopes data
- Role-based ENTITY filtering (e.g., teacher only sees their students) is OUT OF SCOPE for T5.4

## Testing Pattern (Reference)

Each test file follows the same pattern as `cross-school-leakage.test.ts`:

1. **Bootstrap:** Create 2 schools, 1 user with teacher roles at both schools
2. **Seed:** Create students, teachers, classes, subjects at each school
3. **Login:** User logs in once, gets token with roles from both schools
4. **Test:** Switch `x-school-id` header, verify data changes accordingly

Example test structure:

```typescript
it('should return School A students when x-school-id=A', async () => {
  const res = await app.inject({
    method: 'GET',
    url: '/api/v1/students',
    headers: {
      authorization: `Bearer ${tokenMulti}`,
      'x-school-id': String(schoolAId),
    },
  })
  const body = JSON.parse(res.payload)
  expect(body.data).toBeDefined()
  body.data.forEach((student: any) => {
    expect(student.school_id).toBe(schoolAId)
  })
})

it('should return School B students when x-school-id=B', async () => {
  const res = await app.inject({
    method: 'GET',
    url: '/api/v1/students',
    headers: {
      authorization: `Bearer ${tokenMulti}`,
      'x-school-id': String(schoolBId),
    },
  })
  const body = JSON.parse(res.payload)
  expect(body.data).toBeDefined()
  body.data.forEach((student: any) => {
    expect(student.school_id).toBe(schoolBId)
  })
})
```

## Notes

- **Users are global** — school context comes from `user_roles`, not `users` table
- **`x-school-id` header** overrides JWT payload context for each request
- **Multi-role user** = same user ID, multiple `user_roles` entries with different `school_id` values
- **No role-based entity filtering** — teacher role doesn't restrict which students are visible; only `school_id` filtering applies
- Each test file is intentionally small (~50-80 lines) to keep them focused and easy to debug
